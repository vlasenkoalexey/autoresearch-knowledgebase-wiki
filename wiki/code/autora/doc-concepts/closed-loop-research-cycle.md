# The closed-loop empirical research cycle

## Definition
AutoRA's whole reason for existing is one repeating cycle, which `docs/index.md` names the
**Autonomous Empirical Research Paradigm**: "a dynamic interplay between two artificial agents. The
first agent, a theorist, is primarily responsible for constructing computational models by relying on
existing data to link experimental conditions to dependent measures. The second agent, an
experimentalist, is tasked with designing follow-up experiments that can refine and validate the
models generated by the theorist." A third role, the **experiment runner**, sits between them and is
what makes the loop *empirical* rather than simulated: "To enable closed-loop empirical research,
AutoRA interfaces with platforms for automated data collection, such as Prolific or Amazon Mechanical
Turk, which enable the efficient acquisition of behavioral data from human participants."

`pyproject.toml` that pins together roughly twenty separately-published, separately-hosted sibling

`autora-experiment-runner-*`, â€¦) under one importable `autora` distribution. It contains essentially no

modules, and all of them are either mkdocs documentation-site generation helpers

this repo that is directly grounded in the cycle: it is the executable definition of which
sub-namespaces ship by default with a bare `pip install autora` â€” `autora.experiment_runner.synthetic`,
`autora.experimentalist`, `autora.utils`, `autora.variable`, and `autora.workflow` â€” i.e. enough to run
a synthetic (in-silico) version of the closed loop out of the box, while every named theorist
(BMS/BSR/DARTS), every non-default experimentalist, and every real-world experiment runner
(Firebase/Prolific) is an opt-in extra pulled from a *different* repo. The README's worked example
(`sin(x)` discovery) makes the shape of one iteration concrete: `s = experimentalist(s, ...)` â†’
`s = experiment_runner(s, ...)` â†’ `s = theorist(s)`, repeated for a fixed number of cycles, threading a
single [`StandardState`](state-and-delta-data-model.md) object through all three calls.

## Why it matters / when it applies
This is the frame the whole package is built to serve: automating the alternation between theorizing
and experimenting that, per `docs/index.md`'s stated motivation, "is constrained by the rate at which
scientists can alternate between the development of formal theories and the execution of experiments."
It is a **physical/behavioral** closed loop, not an in-silico self-play loop â€” the experiment-runner
step can hand conditions to a real human participant on Prolific or Mechanical Turk and wait for
genuine behavioral data before the theorist ever sees it, which is a materially different kind of
"experiment" than the other autoresearch silos in this wiki run (they close the loop over training
runs, code edits, or simulated environments â€” see the up-link to
[`closed-loop-experiment-design`](../../../concepts/closed-loop-experiment-design.md)).
