# Browser dashboard export — the interactive progress page and the typed lineage graph

## Definition
The README's "Watching progress" section describes three live views of increasing detail: an always-on
terminal **widget**, a **fullscreen overlay** (Alt+G / `research_status`), and **browser dashboards**
(Alt+O / `/research-open` / the `export_dashboard` tool). The browser tier writes two self-contained HTML
files into `.autoresearch-vkf/session/`: `progress.html`, an interactive vanilla-JS dashboard (multi-metric
chart, search-tree view, embedded knowledge graph, coverage heatmap, belief bars, filterable experiment table,
light/dark toggle) that reads a `data.json` sidecar and re-renders in place so an open tab tracks a live run
without losing filters or scroll; and `dashboard.html`, the richer typed idea-lineage graph (with
conflict/derived-from edges) generated by `vkf html`.

## In pi-autoresearch-vkf (grounded)
`progress.html`'s payload is [`buildDashboardData`](../concepts/extensions-pi-autoresearch-vkf-progress_data.ts.md),
already grounded on its own concept page — this doc-concept covers the rendering layer the README describes
but that page doesn't: [`renderDashboardHtml`](../catalog/extensions/pi-autoresearch-vkf/progress_html.ts.md#renderDashboardHtml)
in `progress_html.ts` emits the self-contained document (no build, no dependencies), with
[`embedJson`](../catalog/extensions/pi-autoresearch-vkf/progress_html.ts.md#embedJson) safely inlining the
`data.json` payload into a `<script>` tag and [`escapeHtml`](../catalog/extensions/pi-autoresearch-vkf/progress_html.ts.md#escapeHtml)/[`zoomStage`](../catalog/extensions/pi-autoresearch-vkf/progress_html.ts.md#zoomStage)
handling the pan/zoom search-tree and knowledge-graph panels the README calls out. The client-side application
logic lives in `progress_app.ts` (the tabbed Overview/Experiments/Tree/Knowledge/Plan SPA that fetches the
sidecar and re-renders — kept as arrays of double-quoted JS lines per `CLAUDE.md`'s convention so its own
backticks/`${}` don't clash with the module's template strings) and its design-token CSS custom properties in
`progress_styles.ts`; both are catalog-only modules (no dedicated concept page) since their content is mostly
generated JS/CSS strings rather than TypeScript logic. `dashboard.html`'s heavier typed lineage — with conflict
edges the CLI-free [`buildLineage`](../concepts/extensions-pi-autoresearch-vkf-progress_data.ts.md) doesn't
compute — comes from the separate `vkf html` export path through
[the `vkf` CLI bridge](../concepts/extensions-pi-autoresearch-vkf-vkf.ts.md).

## Why it matters / when it applies
This is the highest-fidelity view of a continuous, unattended run: the terminal widget is necessarily terse
(bottom-truncated, three-bucket memory summary — see
[the terminal dashboard's design rationale](../concepts/extensions-pi-autoresearch-vkf-dashboard.ts.md)), while
`progress.html` has room for the full search tree, the full seven-state memory lifecycle, and a live-updating
multi-metric chart. The "re-renders in place" property — reading a `data.json` sidecar rather than a
`<meta refresh>` whole-page reload — is what the [`progress_data.ts` concept page](../concepts/extensions-pi-autoresearch-vkf-progress_data.ts.md)
frames as the reason data (this module) is split from rendering (`progress_html.ts`/`progress_app.ts`): the
HTML shell is written once, and only the JSON payload changes on every state update.
