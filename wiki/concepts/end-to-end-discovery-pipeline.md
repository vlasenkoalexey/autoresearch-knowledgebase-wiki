# End-to-end discovery pipeline

**Definition.** An end-to-end discovery pipeline is an automated system that carries a research effort through
every stage from hypothesis to publishable artifact without a human performing the scientific content
generation at any stage: idea generation → experiment design and execution → data analysis and visualization →
manuscript writing → peer review, chained so the output of each stage is consumed directly by the next.

**How The AI Scientist-v2 implements it.** [The AI Scientist-v2](../sources/ai-scientist-v2.md) (Sakana AI,
arXiv:2504.08066) chains six phases (source, p.3 Figure 1, p.4 §3): (1) **idea generation** — open-ended,
grant-proposal-style ideation grounded by live Semantic Scholar novelty checks, followed by idea scoring and
archiving (source, p.4 §3.1); (2) **tree-based experimentation** via
[agentic tree search](agentic-tree-search.md) across the manager's four stages (source, p.4–7 §3.2); (3)
**dataset loading**, preferentially through Hugging Face's one-line `datasets.load_dataset`, flagged by the
authors as "somewhat ad-hoc" since not every dataset repo supports it (source, p.7 §3.3); (4) **plotting with
VLM feedback**, where a Vision-Language Model critiques every generated figure during the tree search itself
(source, p.7 §3.4); (5) **manuscript writing** — a single-pass generation (replacing v1's incremental,
Aider-based editing loop) followed by a separate reflection stage powered by a reasoning model (o1), which is
also given the target page limit and current compiled PDF length so it can self-trim to fit submission
constraints (source, p.4, p.10); and (6) **LLM/VLM paper reviewing**, including a second VLM pass that checks
figure/caption/text alignment and flags duplicated figures across the main text and appendix (source, p.7–8
§3.4). The pipeline was run fully autonomously — no human edited code, results, figures, or manuscript text of
the selected final submission — to produce three complete manuscripts submitted blind to the ICLR 2025 ICBINB
workshop; one scored 6.33 average (6, 6, 7) and cleared the workshop's acceptance bar (source, p.8–9 §4,
p.9 point 3).

**Why it matters.** This is the empirical demonstration this wiki cites for "can a fully autonomous pipeline's
*output artifact* — not just its experiments — survive an external, human-run quality gate": the paper reports
the first instance of a wholly AI-generated manuscript passing a real peer-review process (source, p.1 abstract,
p.9). But the same paper's own annotated internal review is the caution attached to that claim: workshop
acceptance rates (60–80%) are structurally much higher than main-conference rates (20–30% at ICLR/ICML/NeurIPS),
and the authors state plainly that the system "does not yet consistently reach the rigorous standard required
for top-tier conference publications, nor does it even reach workshop-level consistently" (source, p.13 §5) —
and a code-level audit of the accepted paper found a mismatch between the described regularizer and what the
code actually computed, plus ~57% train/test overlap the pipeline itself never flagged (source, p.11, p.42–43).

## Grounded in code
The six-phase pipeline above is grounded in the ingested [ai-scientist-v2](../code/ai-scientist-v2/overview.md)
source. Its literal realization is one linear driver script,
[launch_scientist_bfts.py](../code/ai-scientist-v2/concepts/launch_scientist_bfts.md), which chains the stages
end to end: materialize one idea (from [ideation](../code/ai-scientist-v2/concepts/ai_scientist-perform_ideation_temp_free.md))
→ hand off to the tree-search stage (the
[AgentManager](../code/ai-scientist-v2/concepts/ai_scientist-treesearch-agent_manager.md)/[ParallelAgent](../code/ai-scientist-v2/concepts/ai_scientist-treesearch-parallel_agent.md)
[agentic-tree-search](agentic-tree-search.md) engine) → aggregate plots → drive a bounded-retry
[manuscript writeup](../code/ai-scientist-v2/concepts/ai_scientist-perform_icbinb_writeup.md) → run an LLM+VLM
review pass → kill child processes and exit. A verified implementation detail: the writeup's page-limit check
compiles the *actual* PDF each reflection round and measures it (via `pdftotext`) rather than trusting the
model's own token count — the compiler is the oracle.
